● Honest answer: thin frontend over SOPS is fine. Don't build a Vault clone.

  Where the line sits

  Building a secrets manager from scratch is one of the classic "looks easy,
  isn't" traps — auth, session, audit, rotation workflows, encrypted-at-rest,
  revocation semantics. That's months of security-critical code.

  But building a TUI + web UI that wraps SOPS is a weekend project because all
  the hard parts are already solved:

  - Crypto: SOPS + age (don't touch)
  - Auth: Authelia OIDC + forward-auth (don't touch)
  - Storage: git (don't touch)
  - Authorization: lldap group membership, read via LDAP
  - Audit: git log on the dotfiles repo

  Your tool becomes ~1000 lines of Rust: list keys, show masked values, edit →
  re-encrypt → commit → push, rotate with overlap window.

  MVP scope worth building

  secret-tui
    list              # parse secrets/api-keys.yaml, show paths
    show <path>       # sops -d in memory, mask, render
    edit <path>       # edit in $EDITOR → sops -e → git commit → git push
    rotate <path>     # generate new, write, keep old in rotation/ for N days
    pull              # git pull && re-decrypt
    login             # Authelia OIDC device flow (your vault.rs TODO)

  secret-web (axum + htmx, behind Authelia forward-auth)
    same operations, same SOPS calls under the hood

  Both binaries shell out to sops -d / sops -e — zero custom crypto. Authz = "is
   user in admin group in lldap?". Done.

  What you don't build

  - Your own key hierarchy (age handles it)
  - Your own session tokens (Authelia handles it)
  - Dynamic/leased secrets (that's OpenBao's job, and you don't need it)
  - Policies language (lldap groups + file-path prefixes are enough at your
  scale)

  Verdict

  Your secret-tui-rust already exists and has the vault.rs Authelia-OIDC TODO.
  Finish that, add a sibling secret-web-rust crate in the same workspace, share
  the SOPS/git/lldap modules. Authelia's forward-auth means the web side needs
  zero auth code — Traefik injects Remote-User / Remote-Groups headers.

  This is the right scope. Anything bigger, use OpenBao; anything smaller, just
  sops edit at the CLI.

  SOPS assumes a human. Agents don't fit cleanly.

  With pure SOPS:
  - Every MCP server needs its own age identity
  - Adding a new agent = re-encrypt every file to include its pubkey
  - Revoking an agent = re-encrypt + re-generate any secret it saw
  - No audit of "which agent read which key when"

  - Anthropic/OpenAI API keys go into env once at process start

  buy you much yet.

  Where it would buy you something

  If you start building MCPs that:
  - Run untrusted code (code interpreters, sandboxed agents)
  - Need per-tenant scoping (friend's agent can't see your Mistral key)
  - Need audit ("agent X requested cerebras_key at 03:14")

❯ can my age be handled bu aurelia?                                             

● No — and that's a layering confusion worth untangling.

  Authelia ≠ secret store

  Authelia does authentication (who are you, are you logged in, MFA). It has no
  key-value secret API, no file storage endpoint, no decryption service. You
  can't hand Authelia your age key for safekeeping.

  The two layers are complementary, not substitutable

  ┌────────────────────────┬────────────────────────────────────┬───────────┐
  │         Layer          │                Job                 │ Your tool │
  ├────────────────────────┼────────────────────────────────────┼───────────┤
  │ AuthN/AuthZ            │ "Who is this request from?"        │ Authelia  │
  ├────────────────────────┼────────────────────────────────────┼───────────┤
  │ Cryptographic material │ "What can decrypt the ciphertext?" │ age key   │
  └────────────────────────┴────────────────────────────────────┴───────────┘

  Authelia gates who can ask the secret-tui/web to decrypt. The age key is what 
  does the decryption. If Authelia held the key, compromising Authelia =
  compromising every secret — exactly what you're trying to avoid.

  Realistic architecture

  TUI (runs as you on your laptop):
  - Age key at ~/.config/sops/age/keys.txt (laptop-local, OS-level perms)
  - No Authelia involved — you're already authenticated to your OS

  Web (runs as a service on flow.hugo.dk):
  - Service owns its own age key at e.g. /var/lib/secret-web/age.key, readable
  by its systemd unit only
  - Authelia forward-auth decides whether the user can invoke the service
  - Service uses its age key to decrypt; hands back scoped plaintext via HTTP

  Two distinct credentials: user's session (Authelia) + service's decryption key
   (age on-disk).

  What Authelia can help with

  - Unlock the TUI: OIDC device flow → prove you're you before the TUI will even
   try to call SOPS (your vault.rs TODO)
  - Gate the web: forward-auth header on Traefik → web only sees authenticated
  requests
  - MFA on sensitive ops: Authelia's two_factor policy on /rotate, /delete
  routes — you already use this pattern for auth.hugo.dk/admin

  If you want hardware-backed age

  - age-plugin-yubikey — private key lives on YubiKey, touch-to-decrypt
  - age-plugin-fido2-hmac — any FIDO2 device, cheaper
  - Pairs nicely with Authelia's WebAuthn: same hardware, two different trust
  chains

  Bunker pattern for age (mirrors your SSH rotation)

  - Personal age key on laptop (rotate yearly)
  - Bunker age key in a safe (never rotates; recovery-only)
  - Service age keys on each host (rotate with hostname ownership)
  - All three as recipients in .sops.yaml — you already do this with 3 age
  pubkeys

  So: keep age as the crypto primitive, use Authelia for session/MFA at the UI
  layer. They're meant to stack, not replace each other.

  webauthn:

    display_name: "hugo.dk"

  two_factor policy on /admin then resolves to touch-the-YubiKey, which is what

  ┌─ Log into secret-web ─────────────────────────┐
  │  Authelia forward-auth  → passkey (YubiKey)   │
  └───────────────────┬───────────────────────────┘
                      ↓  (you are who you say)
  ┌─ secret-web decrypts ─────────────────────────┐
  │  sops -d api-keys.yaml → prompts YubiKey     │
  │                          (or service age key) │
  └───────────────────┬───────────────────────────┘
                      ↓
                plaintext value
